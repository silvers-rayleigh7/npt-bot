# Урок с опорой на источники (Retrieval-Augmented Lesson) — дизайн

**Дата:** 2026-07-13
**Статус:** согласовано с заказчиком (архитектура одобрена в чате)

## Цель

Усилить генерацию урока (`/api/lesson`) так, чтобы основная модель (GigaChat) перед ответом
**на лету обращалась к источникам знаний** и опиралась на них, а не отвечала только «из головы».
Урок становится привязанным к школьной программе и подкреплённым фактами с указанием источника.

## Ключевой принцип: retrieval-on-demand (не пред-загрузка корпуса)

Мы НЕ заливаем всю систему образования РФ в свою базу. В момент запроса пайплайн
**идёт в нужные места и приносит релевантное**. Три источника = три ретривера, подключённых
к уже работающему пайплайну урока. Это дешевле, законно и не устаревает (веб-факты свежие).

**Инвариант безопасности:** retrieval augment-ит, но НИКОГДА не блокирует. Если все ретриверы
вернули пусто или упали по таймауту — урок всё равно генерируется как сейчас (MVP не ломается).

## Источники (ретриверы)

| Ретривер | Что приносит | Хранилище |
|----------|--------------|-----------|
| **StorylineRetriever** | релевантные из наших 42 сюжетов (semantic match по теме/предмету/региону) | `storylines_index.json` (эмбеддинги, собирается из `content/storylines/`) |
| **ProgramRetriever** | раздел/темы школьной программы для класса+предмета + содержание темы из учебников (структура, термины, примеры) | `programs_index.json` (курируется, общедоступные ФГОС + учебники) |
| **WebRetriever** | 1–2 проверенных научпоп-факта по теме + URL источника | on-demand поиск (внешний search API, за флагом) |

Учебники Минпросвещения — общедоступный образовательный ресурс (решение заказчика), используем свободно.

## Архитектура

```
Форма учителя → LessonIn (grade, subject, topic, location, season)
      │
      ▼
retrieval.gather_context(query, timeout)         # параллельно, ThreadPoolExecutor
   ├─ StorylineRetriever.retrieve(query) → [Snippet]
   ├─ ProgramRetriever.retrieve(query)   → [Snippet]
   └─ WebRetriever.retrieve(query)        → [Snippet]
      │  merge + ранжирование по score, топ-N, лимит символов
      ▼
   блок «Материалы для опоры» (текст)
      │
      ▼
GigaChat(lesson_prompt + опора) → урок
```

Всё в существующем сервисе `site-bar` (`/opt/tropa-bot/server`). Новых сервисов/точек отказа нет.

## Контракты (единые для всех ретриверов)

```python
# base.py
from dataclasses import dataclass, field

@dataclass
class RetrievalQuery:
    grade: str = ""
    subject: str = ""
    topic: str = ""
    location: str = ""
    season: str = ""
    text: str = ""          # свободный текст (topic + subject + …) для матча

@dataclass
class Snippet:
    source: str             # "сюжет" | "программа" | "веб"
    title: str
    text: str               # короткий релевантный фрагмент
    url: str = ""           # ссылка на источник (для веба/сюжета)
    score: float = 0.0      # 0..1, релевантность

class Retriever(Protocol):
    name: str
    def retrieve(self, q: RetrievalQuery, limit: int = 3) -> list[Snippet]: ...
```

`gather_context(query, per_retriever_timeout=6.0, total_char_budget=2500) -> str`
собирает Snippet-ы всех включённых ретриверов, ранжирует, форматирует в markdown-блок:

```
Материалы для опоры (используй по делу, факты цитируй с источником):
[сюжет] Солнечные часы — как тень показывает время …
[программа] 6 класс, география, раздел «Ориентирование» …
[веб] Пчела посещает ~2000 цветков/день — источник: …
```

## Флаги (env, безопасная деградация)

- `RETRIEVAL_ENABLED=1` — мастер-выключатель (0 → урок как сейчас).
- `RETRIEVER_STORYLINES=1`, `RETRIEVER_PROGRAM=1`, `RETRIEVER_WEB=0` — по-ретриверно.
- `RETRIEVAL_TIMEOUT=6` — таймаут на ретривер (сек).
- `WEB_SEARCH_API_KEY=` — ключ поиска для WebRetriever (пусто → веб-ретривер молча выключен).

WebRetriever включаем последним и только если задан ключ — чтобы отсутствие поиска
никогда не роняло урок.

## Эмбеддинги

GigaChat Embeddings API (та же авторизация, что уже есть): модель `Embeddings`.
`embeddings.py` — клиент `embed(texts) -> list[vector]` + `cosine(a, b)`.
Индекс сюжетов считается офлайн (`build_indexes.py`) и коммитится как `data/storylines_index.json`,
чтобы бэкенд не считал эмбеддинги 42 сюжетов на старте. Fallback при недоступности эмбеддингов —
keyword-матч по тегам/заголовкам (ретривер сюжетов не должен падать).

## Порядок сборки

1. **Общие (сам, последовательно):** `base.py`, `embeddings.py` — интерфейсы, от которых зависят агенты.
2. **Параллельно (агенты, изолированные файлы):** `storyline_retriever.py` (A), `program_retriever.py` (B), `web_retriever.py` (C). Друг друга не трогают.
3. **Данные:** `build_indexes.py` + сид `programs_index.json` (стартовый набор: классы/предметы формы учителя).
4. **Интеграция (сам, аккуратно, с проверкой):** `retrieval/__init__.py`, `app.py`, `lesson_prompt.md`. Shared-файлы живого урока — правлю сам, по шагам, за флагом.
5. **Деплой по инкрементам:** сперва с `RETRIEVAL_ENABLED=0` (ничего не меняется), потом включаю по-ретриверно, проверяя урок после каждого.

## Тестирование

- Юнит: каждый ретривер на синтетическом запросе возвращает валидные Snippet (или пусто, без исключений).
- `gather_context`: при падении одного ретривера остальные отрабатывают; при всех пустых → "".
- Интеграция: `/api/lesson` с `RETRIEVAL_ENABLED=0` == текущее поведение (регресс-инвариант).
- Интеграция: с включёнными ретриверами урок содержит опору и цитирует источник для веб-фактов.

## Вне scope (следующий инкремент)

PDF-презентация урока (оформленный скачиваемый артефакт). Делаем после того, как retrieval заработает.

## Риски и снятие

- **Латентность урока** ↑ из-за retrieval → параллельный запуск + жёсткий таймаут на ретривер + бюджет символов.
- **Веб-факты неточны** → WebRetriever приносит только с источником; в промпте — «если факт из веба, укажи источник; сомнительное не включай».
- **Поломка живого урока** → мастер-флаг + инвариант «retrieval не блокирует» + поэтапный деплой.
