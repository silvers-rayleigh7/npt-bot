# Гео-режим на сайте «Тропа» — дизайн

**Цель:** сайт умеет отвечать с учётом местоположения пользователя — как это уже делает
Telegram-бот. Унификация: сайт и бот дают согласованные ответы на «что рядом», потому что
используют один и тот же гео-код (`geo_match.py` + `geo_place.py`), наведённый на один и тот же
контент (сюжеты байт-в-байт).

**Стек:** FastAPI (`site-bar/server`), GigaChat-2-Max → OpenRouter fallback; фронтенд — ванильный
JS-виджет (`site/assets/site-bar/bot.js`) + Leaflet-карта (`templates/route.html`); гео-данные —
Nominatim + Overpass (бесключевые, как у бота).

## Общие ограничения

- Геолокация **только по действию пользователя** (кнопка), не автозапуск. HTTPS уже есть.
- Координаты уходят на серверы OpenStreetMap (Nominatim/Overpass) — предупреждаем строкой в UI.
- Внешние гео-сервисы **не должны ронять ответ**: `build_geo_prompt` уже обёрнут try/except + кэш;
  фронтенд обрабатывает отказ/таймаут разрешения мягко.
- Новых серверных зависимостей нет (`geo_place`/`geo_match` — stdlib).
- Правило единого источника контента не затрагивается: гео-код — это код, не сюжеты.

## Унификация: как один гео-код обслуживает сайт и бот

Канон гео-кода — в боте (`tropa-bot/tg-bot/{geo_place,geo_match}.py`). Сайту нужна идентичная
копия. Механизм — как у зеркала сайта: `sync-geo.sh` копирует оба файла в `site-bar/server/geo/`.
Один код + идентичный контент сюжетов (уже гарантирован генератором из базы) ⇒ сайт и бот находят
одни и те же ближайшие сюжеты и одинаково описывают местность. Дрейф исключён: расхождение копий
ловится сверкой в `sync-geo.sh`.

## Фаза 0 — общий гео-фундамент (бэкенд)

**Файлы:**
- Создать `site-bar/server/geo/geo_place.py`, `site-bar/server/geo/geo_match.py` (вендор из бота).
- Создать `tools/sync-geo.sh` (копирует канон → `site-bar/server/geo/`, сверяет идентичность).
- Изменить `site-bar/server/app.py`: эндпоинт `POST /api/geo`.

**Эндпоинт `POST /api/geo`:**
- Вход: `{lat: float, lon: float, question?: str, page_context?: str, history?: [...]}`.
- Логика: `prompt = build_geo_prompt(lat, lon, STORYLINES_DIR)`; собрать messages как в `/api/chat`,
  но добавить `[ГЕО]`-блок; `generate()`.
- Выход: `{answer, provider, place: {name}, nearby: [{title, slug, url, dist_m}]}`.
  `nearby` строится из `nearest_storylines(lat, lon, STORYLINES_DIR, top=3)`; `slug` = имя файла,
  `url = f"{SITE_URL}/storylines/{slug}/"`.
- `STORYLINES_DIR` — env (дефолт `../content/storylines` относительно сервера или абсолют на боксе).

**Правка `geo_match`/`nearest_storylines`:** добавить в возврат `slug` (basename файла без `.md`),
чтобы сайт давал ссылки. Бот `slug` игнорирует — обратная совместимость сохранена.

**Доступ бэкенда к координатам сюжетов (деплой):** сайт деплоится как статика в `/var/www/tropa/site`,
а бэкенд живёт в `/opt/tropa-bot/server` — markdown-сюжетов у него может не быть. `/api/geo` нужны
только `geo:`-координаты + заголовок + slug. Решение: `build.py` (или `sync-geo.sh`) генерирует
лёгкий индекс `site-bar/server/geo/geo_index.json` (`[{slug, title, lat, lon}]`) из frontmatter
сюжетов; `nearest_storylines` на сайте читает индекс, а не сканирует `.md`. Индекс деплоится вместе
с бэкендом. Это и быстрее (не парсим markdown на каждый запрос), и снимает зависимость от наличия
контента рядом с сервером.

## Фаза 1 — гео-чат в виджете

**Файлы:** `site/assets/site-bar/bot.js`, `site/assets/site-bar/bot.css`.

- Чип «📍 Рядом» среди стартовых `CHIPS`. По клику — `navigator.geolocation.getCurrentPosition`.
- Успех: координаты в переменную сессии `geo = {lat, lon}`; показать строку приватности
  («ищу ближайшие точки; координаты уходят на OpenStreetMap»); отправить на `/api/geo`.
- Пока `geo` задан — вопросы идут на `/api/geo` (иначе `/api/chat`), координаты прикладываются.
- Ответ рендерит `nearby` кликабельными ссылками на страницы сюжетов.
- Отказ/таймаут/нет поддержки → мягкое сообщение, обычный чат продолжает работать.

## Фаза 2 — карта на пользователе

**Файлы:** `templates/route.html` (Leaflet-карта), возможно `templates/base.html`.

- Контрол «📍 Я здесь» на карте → `navigator.geolocation` → центр на пользователе + маркер
  «вы здесь» + маркеры ближайших сюжетов с расстояниями (данные — из `/api/geo` или клиентский
  haversine по уже отрисованным точкам).

## Фаза 3 — маршрут «от меня»

**Файлы:** `templates/route.html` (существующая интеграция Valhalla/TSP).

- «Маршрут от меня» к выбранному сюжету: стартовая точка = геолокация пользователя, дальше —
  существующая маршрутизация Valhalla.

## Порядок и тестирование

Пофазно: **0+1 → 2 → 3**, деплой и проверка на бою между фазами.

- Фаза 0: `curl -X POST /api/geo` с координатами Иннополиса и удалённой точки — проверить `nearby`,
  `place`, что внешний таймаут не роняет ответ.
- Фаза 1: клик «Рядом» → разрешение → ссылки на сюжеты кликабельны; отказ в разрешении → чат жив.
- Фазы 2–3: маркер и центр карты; маршрут от произвольной точки.
- Триада проверки (правило проекта): программно (curl/JS), визуально (скриншот), симуляция клика.
